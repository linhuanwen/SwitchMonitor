# SwitchMonitor 多站组网架构设计

> 状态：设计完成，待实现
> 日期：2026-07-19

---

## 1. 物理拓扑

```
                         ┌──────────────┐
                         │  总监测终端    │
                         │ (可看所有站)   │
                         └──┬──┬──┬──┬──┘
                            │  │  │  │
              ┌─────────────┘  │  │  └─────────────┐
              ▼                ▼  ▼                ▼
    ┌─────────────┐  ┌─────────────┐  ┌─────────────┐
    │ 站机 A (SSB) │◄─┤ 站机 B (DHD) │◄─┤ 站机 C (MDZ) │
    │ 班组α 成员   │  │ 班组α 成员   │  │ 班组α 成员   │
    └─────────────┘  └─────────────┘  └─────────────┘
         │   ▲             │   ▲             │   ▲
         └───┼─────────────┘   │             │   │
             └─────────────────┼─────────────┘   │
                               └─────────────────┘
                              同班组全互联推送
```

**三种角色：**

| 角色 | 说明 | 数据流向 |
|------|------|---------|
| **站机** | 机房里跑 CSM + SwitchMonitor 的工控机。产生本站数据，也作为班组查看终端 | 推送本站数据给同班组所有站机 + 总终端；接收同班组其他站的推送 |
| **班组终端** | 跟站机是同一台物理机。通过配置限定可查看的站点列表 | 查看班组管辖 1~3 个站的数据 |
| **总终端** | 监测中心，独立的工控机。可查看所有站点 | 接收所有站的推送，可手动补拉任意站点数据 |

**关键设计：同一班组内各站机全互联推送，无中转。** 一台站机挂了不影响同组其他站机互通。

## 2. 通信协议

**HTTP + JSON + gzip，.NET 4.0 内置，零第三方依赖。**

### 2.1 站机端接口（DataForwarder.exe 提供）

| 端点 | 方法 | 用途 | 调用方 |
|------|------|------|--------|
| `/api/status` | GET | 健康检查 + 获取最新数据时间戳 | 总终端 / 班组终端（2分钟一次） |
| `/api/events?since={timestamp}` | GET | 补拉指定时间戳之后的所有增量数据 | 总终端 / 班组终端（自动或手动触发） |

**GET /api/status 响应：**
```json
{
  "stationId": "SSB",
  "stationName": "三水北站",
  "status": "ok",
  "lastTimestamp": 1712345678,
  "dbSizeMB": 320
}
```

**GET /api/events?since=1712340000 响应（gzip 压缩）：**
```json
{
  "stationId": "SSB",
  "since": 1712340000,
  "count": 5,
  "events": [
    {
      "switchId": "1-J",
      "timestamp": 1712340005,
      "direction": "定位到反位",
      "durationSec": 11.76,
      "sampleInterval": 0.04,
      "currentA": [[0.0, 0.5], [0.04, 0.8], ...],
      "currentB": [[0.0, 0.3], [0.04, 0.6], ...],
      "currentC": [[0.0, 0.4], [0.04, 0.7], ...],
      "power": [[0.0, 0.1], [0.04, 0.5], ...],
      "diagnosis": { "level": "正常", "results": [] }
    }
  ]
}
```

### 2.2 总终端/班组终端接收端点

| 端点 | 方法 | 用途 | 调用方 |
|------|------|------|--------|
| `/api/receive` | POST | 接收站机主动推送的增量数据包 | 站机 DataForwarder |

**POST /api/receive 请求体（gzip 压缩）：**
```json
{
  "stationId": "SSB",
  "batchTimestamp": 1712345678,
  "events": [ ... ]   // 结构与 /api/events 响应中的 events 相同
}
```

### 2.3 推送时机

```
道岔动作 → SwitchMonitor 写入 SQLite → DataForwarder 下一次轮询（≤1秒）检测到
    → 等待 1 秒合并窗口 → gzip 打包 → POST 给所有订阅者
    → 从动作产生到总终端可见 ≤ 10 秒
```

### 2.4 重试策略

推送失败 → 重试 3 次（间隔 2s / 4s / 8s）→ 仍失败则放弃。等对方恢复后由自动补拉兜底。

## 3. 活性检测与数据补拉

### 3.1 主动探测

总终端 / 班组终端每隔 **2 分钟** 向所有订阅的站机发送 `GET /api/status`，**超时 10 秒**。

```
在线 → 绿色
离线（连续 2 次探测失败）→ 红色 + 弹提示"XX站通信中断"
恢复 → 自动触发补拉
```

### 3.2 自动补拉

探测到站机从"离线"变为"在线"时自动触发：

```
GET /api/events?since={lastTimestamp}
→ 按时间升序返回 → 边收边写入 SQLite → 全程不更新 lastTimestamp
→ 全部完成后 → 持久化 lastTimestamp
```

中途失败不更新标记，下次重拉从同一 `since` 开始，宁可重复不遗漏。

### 3.3 手动补拉

总终端 UI 上提供"补拉数据"按钮，手动触发一次自动补拉逻辑。

## 4. 数据存储（SQLite）

### 4.1 目录结构

```
parsed_data/
├── SSB.db          ← 三水北站
├── DHD.db          ← 大湖东站
└── MDZ.db          ← 某站
```

每站一个独立 `.db` 文件。切换站点 = 切换数据库连接。

### 4.2 表结构

```sql
CREATE TABLE events (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    switch_id   TEXT    NOT NULL,       -- 如 "1-J"
    timestamp   INTEGER NOT NULL,       -- Unix 秒
    direction   TEXT,                    -- "定位到反位" / "反位到定位"
    duration_sec REAL,
    sample_interval REAL DEFAULT 0.04,
    sample_count  INTEGER,
    current_a   BLOB,                    -- float32 LE 数组
    current_b   BLOB,
    current_c   BLOB,
    power       BLOB,                    -- float32 LE 数组
    diag_json   TEXT,                    -- 诊断结果 JSON
    created_at  TEXT DEFAULT (datetime('now'))
);

CREATE INDEX idx_events_switch_time ON events(switch_id, timestamp);
CREATE INDEX idx_events_timestamp ON events(timestamp);
```

### 4.3 空间估算

| 站型 | 转辙机数 | 日增量 | 年增量 |
|------|---------|--------|--------|
| ZYJ7 小站 | 8 台 | ~0.9MB | ~330MB |
| ZYJ7 大站 | 80 台 | ~13MB | ~4.8GB |
| ZDJ9 中站 | 16 台 | ~3.5MB | ~1.3GB |

### 4.4 数据清理

**手动触发，按站点清理。** 总终端 UI 提供清理功能：

- 选择目标站点（可多选）
- 设置保留天数（默认 365 天）
- 展示当前数据量、最早记录日期、预计释放空间
- 确认后执行 DELETE + VACUUM

站机 / 班组终端不设自动清理（数据量小，硬盘足够）。

### 4.5 已有 JSON 数据迁移

提供一次性导入脚本，将 `parsed_data/` 下现有 JSON 文件导入 SQLite，导入后删除原 JSON。

### 5.2 总终端配置（config.json）

```json
{
  "role": "central",
  "listenPort": 9000,
  "dataRetentionDays": 365,
  "stations": [
    {"id": "SSB", "name": "三水北站", "ip": "192.168.1.10", "port": 9000},
    {"id": "DHD", "name": "大湖东站", "ip": "192.168.1.11", "port": 9000},
    {"id": "CEN", "name": "岑村站",   "ip": "192.168.1.13", "port": 9000}
  ]

```

### 5.3 关键字段说明

| 字段 | 站机 | 总终端 | 说明 |
|------|:---:|:-----:|------|
| `role` | `"station"` | `"central"` | 角色标识 |
| `stationId` | ✅ | - | 本站标识 |
| `vendorType` | ✅ | - | 厂商类型：`"huihuang"` / `"bangcheng"` / `"tonghao"` |
| `listenPort` | ✅ | ✅ | 本地 HTTP 监听端口 |
| `subscribers` | ✅ | - | 推送目标列表（`IP:Port`），不含自己 |
| `teamStations` | ✅ | - | 班组管辖的站点列表 |
| `stations` | - | ✅ | 总终端管理的所有站点列表 |
| `switchGroups[].switchType` | ✅ | - | 每台转辙机的型号（`"ZYJ7"` / `"ZDJ9"`），手动配置 |
| `dataRetentionDays` | 0（不限） | 365 | 数据保留天数，0 = 不限制 |
| `mergeWindowMs` | ✅ | - | 推送打包合并窗口（毫秒） |

## 6. 进程架构

每台工控机上运行两个独立进程：

```
┌─────────────────────────┐
│  SwitchMonitor.exe      │  主程序：解析、特征提取、诊断、UI 展示
│  - 厂商格式解析          │  负责 SQLite 读写
│  - 诊断引擎              │
│  - WinForms UI          │
│  - 数据清理功能          │
└─────────────────────────┘
           │
           │ 共享 SQLite 文件（只读）
           ▼
┌─────────────────────────┐
│  DataForwarder.exe      │  后台转发：监控 SQLite → 推送 → 响应 status
│  - 每 1 秒扫描 SQLite    │  不写 SQLite，只读
│  - 打包 + gzip + POST   │  sync_state 写入独立小文件
│  - GET /api/status      │
│  - GET /api/events      │
│  - POST /api/receive    │
└─────────────────────────┘
```

**同步状态存储：** DataForwarder 使用独立文件 `.sync_state.json` 持久化每个订阅者的最后推送时间戳，避免与 SwitchMonitor 的 SQLite 写入产生锁冲突。

## 7. 厂商适配

`config.json` 中每个站点配置 `vendorType` 字段：

```json
{ "vendorType": "huihuang" }   // 辉煌
{ "vendorType": "bangcheng" }  // 邦诚
{ "vendorType": "tonghao" }    // 通号
```

程序根据此字段选择对应的解析器。`switchGroups[].switchType` 额外指定每台转辙机的机械型号，确保诊断规则使用正确的基线。

## 8. 决策记录汇总

| # | 决策点 | 结论 |
|---|--------|------|
| 1 | 拓扑模型 | 站机同班组全互联推送 + 直连总终端，无中转 |
| 2 | 通信协议 | HTTP + JSON + gzip，.NET 4.0 内置 |
| 3 | 接口 | GET /api/status, GET /api/events, POST /api/receive |
| 4 | 实时性 | 数据产生后 ≤ 10 秒内可查看 |
| 5 | 打包策略 | 1 秒合并窗口，推送完整 SwitchEvent + 诊断结果 |
| 6 | 健康检查 | 2 分钟主动探测，10 秒超时，离线标红告警 |
| 7 | 数据补拉 | 离线恢复后自动触发，手动按钮兜底 |
| 8 | 补拉顺序 | 从旧到新，全完成后才更新标记，中断不丢 |
| 9 | 存储引擎 | SQLite，每站一个 .db |
| 10 | 表结构 | events 单表，4 个独立 BLOB（电流 A/B/C + 功率） |
| 11 | 数据迁移 | 一次性导入脚本，导入后删原 JSON |
| 12 | 数据清理 | 手动触发，按站点清理，默认保留 365 天 |
| 13 | 同步状态 | DataForwarder 用独立 .sync_state.json，不写 SQLite |
| 14 | 重试策略 | 推送失败重试 3 次 → 放弃 → 靠补拉兜底 |
| 15 | 进程模型 | DataForwarder.exe 独立进程，轮询 SQLite 每秒 |
| 16 | 配置管理 | 三种角色（station/central）共用 config.json，字段不同 |
| 17 | 厂商适配 | vendorType 字段 + switchGroups[].switchType 手动配置 |
| 18 | 总终端 UI | WinForms 复用现有组件，按站切换 |
| 19 | 班组终端 | 与站机同一台物理机，通过 teamStations 限定可查看站点 |
| 20 | 全互联范围 | 同班组内所有站机互相推送，不跨班组 |
