## 1. 模块目标

对每次道岔动作的**功率曲线**自动运行报警分析，输出 正常/预警/报警/故障 四级结论，
在 UI 中标记异常动作。分三层：

- **L1 完整性硬规则**（不依赖基线）：超时/夭折/采集异常 → D1-D4
- **L2 分阶段特征阈值**（相对基线的偏差）→ D1-D4
- **L3 逐点对比 + 趋势分析**（二期）→ D6

一期只分析**功率曲线**；三相电流曲线规则留二期。

## 2. 数据实证发现（设计依据）

### 2.1 转辙机转换五阶段在数据中的形态

采样率 25Hz（0.04s/点），单位 kW。以 1-1 (J型) 的 2,962 条正常曲线逐点中位数为例：

| 阶段 | 曲线形态 | 位置与实测值 |
|---|---|---|
| ① 启动 | 先出现 2 点小台阶（0.14→0.24），随后**启动尖峰** | 尖峰固定出现在第 6-7 点（0.24-0.28s），峰值中位 3.24 kW |
| ② 解锁 | 尖峰后回落到略高的小平台，约 0.5s | 均值中位 0.307 kW |
| ③ 转换 | 长平台，占曲线主体 | 均值中位 0.301 kW，段内最大值中位 0.373 kW |
| ④ 锁闭 | 尾部**卸载凹口**（跌到 ~0.16） | activeEnd 前约 22-30 点处 |
| ⑤ 缓放 | 凹口后 ~0.9s 低平台，然后归零 | 均值中位 0.214 kW，约 20-22 点长 |

曲线首尾各有 3-4 个 0 值点（录制窗口前后余量）。

### 2.4 数据中已存在的真实异常（共 23,999 事件）

| 异常类型 | 数量 | 曲线形态 |
|---|---|---|
| 打满 790 点录制窗口（≈31.6s） | 180 | 转换平台抬高到 ~0.55-0.66 kW 后一直持续到窗口结束，个别中途有二次启动尖峰 → 典型**卡阻/摩擦连接器空转** |
| 动作夭折（时长 < 中位 60%） | ~24 | 启动后双尖峰冲到 4.5 kW 随即断电（0.8-4.4s 终止）→ **未能解锁/保护切断** |

4# 道岔（4-1/4-X）异常占比最高（3.17% / 2.27%），1-X / 3-X 最干净（0.70% / 0.53%）。

### 2.5 特征漂移（决定基线更新策略）

按月统计（2025-12 → 2026-06，冬→夏）：时长几乎不变（-0.4%），转换段均值缓降约 3%，
启动峰值缓降约 2%。→ **固定基线 + 季度重算已足够**；基线重算做成一键工具（D2）。

## 3. 特征提取与阶段分割算法规格（权威定义）

输入：一次动作的功率采样值数组 `v[0..n-1]`（kW，0.04s/点，从 `SwitchEvent.Power` 的
`[t, value]` 对中取 value 列）。输出 `CurveFeatures`：

```
sampleCount  = n
isFullWindow = (n >= 780)                      # 打满 790 点录制窗口（留余量）
peakAll      = max(v)
isValid      = (n > 0 && peakAll > 0.01)       # false → 采集异常，其余特征不计算

# 有效动作终点（去掉尾部零填充）
threshold    = max(peakAll * 0.05, 0.01)
activeEnd    = 最后一个 v[i] > threshold 的下标 i
durationSec  = round((activeEnd + 1) * 0.04, 2)

# ① 启动尖峰：前 15 点内的最大值
spikePeak    = max(v[0..14]);  spikeIndex = 其下标（多个相同取第一个）

# ② 解锁段：尖峰后 [spikeIndex+2, spikeIndex+14) 共 12 点（0.48s）
unlockMean   = mean(该段)

# ③ 转换段：[spikeIndex+20, activeEnd-40)；若区间无效退化为 [spikeIndex+2, activeEnd)；
#    仍为空则取 v[0..activeEnd]
convMean     = mean(转换段);  convMax = max(转换段)

# 台阶比（检测中途受阻）：转换段等分三份
stepRatio    = mean(后1/3) / max(mean(前1/3), 0.01)    # 前1/3长度<5点时恒为 1.0

# ⑤ 缓放尾段：[activeEnd-22, activeEnd-2) 共 20 点；activeEnd<=30 时为空段，tailMean=0
tailMean     = mean(该段)
```

均值/最大值均保留 3 位小数（`Math.Round(x, 3)`，默认 banker's rounding，与 Python 一致）。

## 4. 报警规则表（默认阈值 + 全量数据演习结果）

级别递增：正常 < 预警 < 报警 < 故障。`ref*` 为该道岔基线值（见 §5）。

| ID | 规则名 | 判据（默认阈值） | 级别 | 对应病害 | 24k 演习触发数 |
|---|---|---|---|---|---|
| R0 | 采集异常 | `!isValid` | 报警 | 采集/解析故障 | 6 |
| R1 | 动作超时/未完成 | `isFullWindow \|\| durationSec > refDur + 3.0` | 故障 | 卡阻、空转、异物 | 198 |
| R2 | 动作夭折 | `durationSec < refDur × 0.6` | 报警 | 未解锁、保护切断 | 24 |
| R3 | 动作时长偏差 | `\|durationSec − refDur\| > 0.5` | 预警 | 阻力变化早期征兆 | 147 |
| R4 | 启动峰值偏高 | `spikePeak > refSpikePeak × 1.3` | 预警 | 启动回路/机械卡滞 | 42 |
| R5 | 转换段功率偏高 | `convMean > refConvMean × 1.3` | 预警 | 滑床板缺油、异物 | 6 |
| R6 | 转换段台阶突变 | `stepRatio > 1.5 \|\| stepRatio < 0.67` | 报警 | 中途受阻/空转 | 1 |
| R7 | 解锁段偏高 | `unlockMean > refUnlockMean × 1.3` | 预警 | 密贴过紧、卡缺口 | 未演习 |
| R8 | 缓放段异常 | `tailMean 偏离 refTailMean ±30%`（tailMean=0 视为缺失） | 预警 | 锁闭/开闭器异常 | 未演习 |

**判定顺序**: R0 命中 → 终止；R1 命中 → 终止；R2 命中 → 终止；否则 R3-R8 全部评估（可多条同时命中）。
事件综合级别 = 命中规则的最高级别；无命中 = 正常。
触发数由 `diag_reference_check.py dryrun` 生成（**验收基准**）。注意 R5/R6 的触发数是在
"R1/R2 命中即终止"语义下统计的——超时/夭折曲线的转换段必然异常，终止评估避免重复报警。

**演习总体结果**: 375/23,999 事件触发（1.56%），按每台每天约 15 动折算 ≈ 每台每天 0.2 条报警，
运维可接受。分道岔触发率：1-1 1.87%、1-X 0.70%、3-1 1.23%、3-X 0.53%、2-1 1.63%、
2-X 1.10%、4-1 3.17%、4-X 2.27%。

## 5. 基线定义

对每台道岔（switchId）分别计算，来源为该道岔全部历史曲线：

```
正常样本筛选: 排除 isFullWindow、durationSec < 2.4s、!isValid；
              取 |durationSec − 时长中位数| < 中位数×15% 的曲线
基线值:       refDurationSec / refSpikePeak / refUnlockMean / refConvMean / refTailMean
              = 正常样本各特征的中位数
元数据:       sampleCount（参与计算的曲线数）、dateFrom/dateTo、computedAt
```

一期**不分动作方向**建基线（现有 Direction 为交替分配的占位值；实测同一道岔时长分布无双峰，
方向差异可忽略）。逐点参考曲线（曲线模板）留 D6。

各道岔基线期望值（D2 验收用，容差 ±0.02）：见 §2.2 表格（含全部 5 项基线值与正常样本数）。

## 7. Slice 划分与依赖

```
D1 特征提取器 + 诊断项目脚手架     ← 可立即开始
D2 基线构建工具                    ← 依赖 D1
D3 规则引擎 + thresholds.json      ← 依赖 D1、D2
D4 管道集成 + 结果存储             ← 依赖 D3
D5 UI 报警展示                     ← 依赖 D4
D6 (二期) 趋势分析 + 逐点对比      ← 依赖 D4，可与 D5 并行
```

新增 C# 项目（进 `SwitchMonitor.sln`，.NET 4.0 / x86，与现有项目一致）：
- `SwitchMonitor.Diagnosis`（类库）：模型 + 特征提取 + 基线 + 规则引擎。依赖 `SwitchMonitor.Data`。
- `SwitchMonitor.DiagTool`（控制台）：selftest / baseline / dryrun 子命令，验收与运维工具。

> 与 PRD 原设计的差异说明：PRD 曾设想 Diagnosis(接口) + DiagnosisEngine(实现) 两个程序集。
> 现简化为单个 `SwitchMonitor.Diagnosis` 程序集，可替换性由 `Rules/*.json` 配置 + 整体换 DLL 保证，
> 减少项目数量。UI 仅通过 `IDiagnosisEngine` 接口调用。

## 8. 待确认 / 开放问题

1. **J/X 与尖轨/芯轨的物理对应**：数据上确凿存在 12s 与 8.7s 两类，但 `1-1`/`1-X` 标签与
   实际尖轨/芯轨的对应需现场核对（同 PRD 既有遗留问题）。**不阻塞本模块**——基线按 switchId 各建各的。
2. **动作方向（定→反/反→定）**：当前为占位交替分配。一期不分方向建基线；现场核对后如需分方向，
   基线结构已按 switchId 组织，扩展为 switchId+direction 即可。
3. **缓放段物理含义**：曲线尾部 0.9s 低平台的确切电气含义（自动开闭器缓放/表示回路）待领域资料核实，
   不影响 R8 的统计学判据。
4. **shiqi 数据集**（另一套采集，1,010 条 .pow，单位 W，~5.6s 短曲线）：形态同样是
   尖峰+平台+尾部小峰，但设备不同，**本模块一期不纳入**，算法设计已兼容（全部为相对基线判断）。
